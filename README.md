# lingyu-chat


## 1. 什么是智能客服？

智能客服是一个基于大语言模型的自动化对话系统，能够理解用户需求、记住历史对话、提取关键信息，并按照业务规则给出回答。

在你的lingyu-chat项目中，智能客服主要帮用户做以下几件事：

**查快递**：用户问“我的订单到哪了”，系统帮用户查询物流进度。

**改地址**：用户说“帮我换个收货地址”，系统帮用户修改配送地址。

**退货退款**：用户说“我要退货”，系统引导用户完成退款流程。

**投诉处理**：用户表达不满时，系统安抚用户并收集问题。

**日常闲聊**：用户聊非业务话题时，系统友好回应。

同时，系统会记住用户说过的订单号、地址等信息，下次再问时不需要重复提供。早期对话会被压缩成摘要，既保留关键信息又控制成本，支持长期连续对话。

## 4. 代码实现

### 4.1. 意图识别

**意图识别**，其实就是让大模型根据我们定好的规则，去读懂用户输入的内容，然后判断出他们到底想干什么。

比如我们的 Prompt 定义如下：

```python
INTENT_RECOGNIZE_PROMPT = """
你是电商客服意图识别器。你必须只输出 JSON，不要输出任何解释文字。
输出 JSON schema：
{{"intents": ["track_shipping"], "slots": {{"order_id": null, "new_address": null}}, "confidence": 0.0}}

候选 intents：
- track_shipping 物流查询
- change_address 改地址
- refund         退货退款
- complaint      投诉
- general        其它/闲聊

规则：
- intents 可包含多个
- confidence 取值 0~1
- 未提到订单号则 order_id 为 null；未提到新地址则 new_address 为 null
"""
```

💡 **举例说明：单个意图**

当用户输入：

> “我的快递单号是123456，帮我查一下到哪了”

经过你的代码处理后，意图识别器会输出类似这样的结构化数据：

```python
{
  "intents": ["track_shipping"],
  "slots": {
    "order_id": "123456",
    "new_address": null
  },
  "confidence": 0.98
}
```

这样，后端程序就可以直接读取这个 JSON，自动调用物流查询接口，而无需人工介入。

💡 **举例说明（多意图 + 缺失信息）**

当用户输入：

> "改地址，另外之前那单也帮我退了"

经过你的代码处理后，意图识别器会输出类似这样的结构化数据：

```python
{
  "intents": ["change_address", "refund"],
  "slots": {
    "order_id": null,
    "new_address": null
  },
  "confidence": 0.78
}
```

注意：用户没有提供订单号和新地址，所以 `slots` 中两个字段都是 `null`，置信度也相对较低，后端程序可以据此决定是否需要追问用户补充信息。

#### 4.1.1. 拓展内容：结构化输出

[模型 | LangChain 中文文档](https://langchain-doc.cn/v1/python/langchain/models.html#结构化输出)

```python
from pydantic import BaseModel, Field

class IntentResult(BaseModel):
    """意图识别结果：通过大语言模型识别用户输入的意图，输出结构化数据。"""
    intents: list[str] = Field(description="意图列表，每个元素为一个意图名称")
    slots: dict[str, Any] = Field(description="slot值字典，键为slot名称，值为slot值")
    confidence: float = Field(description="置信度分数，取值范围0~1，越大表示越确信该意图")
```

#### 4.2.1. 智能客服关键挑战

| 问题                          | 传统做法               | 问题所在                       | 解决方案                     |
| :---------------------------- | :--------------------- | :----------------------------- | :--------------------------- |
| **问题1：如何让AI记住对话？** | 每次请求只发送当前问题 | AI没有上下文，无法理解前后关联 | 每次携带历史消息列表         |
| **问题2：Token成本爆炸**      | 100轮对话携带200条消息 | 成本线性增长，响应变慢         | 长期会话→摘要，近期会话→原文 |
| **问题3：关键信息丢失**       | 摘要压缩时丢失具体信息 | 订单号、地址等无法追溯         | 独立存储关键事实(Facts)      |

#### 4.2.2. 如何做会话隔离？

什么是会话隔离：

```python
┌─────────────────────────────────────────────────────────────────┐
│                        会话隔离问题                               │
├─────────────────────────────────────────────────────────────────┤
│   用户A                    用户B                                 │
│     │                        │                                  │
│     ├── 会话1                ├── 会话1                           │
│     └── 会话2                └── 会话2                           │
│   ❓ 当系统收到"会话1"的请求时，如何区分是用户A的还是用户B的？     │
└─────────────────────────────────────────────────────────────────┘
```

会话隔离的核心问题在于：当多个用户各自创建了同名的会话（比如都叫“会话1”），系统如何区分这些会话分别属于哪个用户。

解决方案是使用**复合唯一标识符**，也就是将用户ID和会话ID组合起来作为存储的Key。例如，用户A的会话1存储为"user_a:session_1"，用户B的会话1存储为"user_b:session_1"。这样即使会话名称相同，由于用户ID不同，它们在存储层是完全隔离的。

这样做的好处是：不同用户的同名会话不会互相覆盖；同一用户的不同会话可以独立管理；系统可以准确地为每个请求找到正确的会话历史。这也是为什么lingyu项目使用`f"session:{user_id}:{session_id}"`作为存储Key的原因——user_id拼接session_id，确保了全局唯一性。

```python
# 存储每个用户的对话历史：key是user_id:session_id，value是会话历史记录
_session_store: Dict[str, BaseChatMessageHistory] = {}
```

#### 4.2.3. 摘要如何生成？

lingyu采用增量摘要机制，每累计 12 条新消息（即 6 轮完整对话，每轮包含1条用户消息+1条AI消息）自动触发一次摘要生成。

具体规则如下：

| 阶段      | 消息累计             | 触发动作       | 摘要内容                             |
| :-------- | :------------------- | :------------- | :----------------------------------- |
| 第1-6轮   | 12条消息             | 触发第1次摘要  | 基于12条消息生成初始摘要             |
| 第7-12轮  | 再增12条（累计24条） | 触发第2次摘要  | 旧摘要 + 新12条消息 → 合并生成新摘要 |
| 第13-18轮 | 再增12条（累计36条） | 触发第3次摘要  | 旧摘要 + 新12条消息 → 合并生成新摘要 |
| ...       | 以此类推             | 每12条触发一次 | 增量合并，永不重复总结               |

##### 4.2.3.4. 生成摘要的代码实现

```python
# 提示词：
SUMMARY_GENERATION_PROMPT = """
你是对话摘要专家。请将【旧摘要】和【新对话】合并成一个新的摘要。

要求：
1. 保留旧摘要中的关键信息（订单号、地址、偏好等）
2. 整合新对话中的重要内容
3. 控制总长度在250字以内
4. 输出格式：[摘要] 内容...

请输出合并后的新摘要：
```

```python
# 生成摘要的代码如下：

# 存储每个用户的对话历史：key是user_id:session_id，value是会话历史记录
_session_store: Dict[str, BaseChatMessageHistory] = {}
# 存储每个用户的会话摘要：key是user_id:session_id，value是会话摘要
_summary_store: Dict[str, str] = {}
# 存储每个用户的未摘要消息数量：key是user_id:session_id，value是未摘要消息数量
_unsummarized_count: Dict[str, int] = {}


def __generate_incremental_summary(
        self,
        old_summary: str,
        new_messages: List[BaseMessage],
        llm=None
) -> str:
    """
    生成增量会话摘要：将[旧摘要]和[新消息]合并后，调用大语言模型生成增量摘要
    :param old_summary: 旧的会话摘要
    :param new_messages: 新的会话历史记录
    :return: 增量会话摘要
    """
    # 如果没有新的消息，直接返回旧的摘要
    if not new_messages:
        return old_summary

    # 提取新消息的文本内容
    new_messages_text = ""
    for message in new_messages:
        if isinstance(message, HumanMessage):
            new_messages_text += f"human：{message.content}\n"
        elif isinstance(message, AIMessage):
            new_messages_text += f"ai：{message.content}\n"

    # 如果没有llm，那么使用简要版本
    if not llm:
        return self._simple_incremental_summary(old_summary, new_messages_text)

    # 使用LLM生成增量摘要
    summary_prompt = ChatPromptTemplate.from_messages([
        ("system", SUMMARY_GENERATION_PROMPT),
        ("human", f"旧摘要：{old_summary}\n新消息：{new_messages_text}")
    ])
    try:
        chain = summary_prompt | llm | str_output_parser
        result = chain.invoke(input={})
        new_summary = result.strip()

        if new_summary.startswith("[摘要]"):
            new_summary = new_summary[7:].strip()
        return new_summary
    except:
        return self._simple_incremental_summary(old_summary, new_messages_text)

def _simple_incremental_summary(
        self,
        old_summary: str,
        new_messages_text: str
) -> str:
    """
    简单的增量会话摘要：将[旧摘要]和[新消息]合并后，返回合并后的摘要
    :param old_summary: 旧的会话摘要
    :param new_messages_text: 新的会话历史记录
    :return: 增量会话摘要
    """
    if old_summary and old_summary != "[摘要] 无":
        # 截取旧摘要的前150字 + 新消息的前150字
        old_part = old_summary[:150] if len(old_summary) >= 150 else old_summary
        new_part = new_messages_text[:150] if len(new_messages_text) >= 150 else new_messages_text
        return f"[摘要] {old_part}...{new_part}"
    else:
        return f"[摘要] {new_messages_text[:150]}"

def _trim_and_summarize(self, llm=None) -> None:
    history = self.get_session_history()
    if not history:
        return

    unsummarized = _unsummarized_count.get(self.__session_key, 0)
    # 达到阈值，触发摘要更新
    if unsummarized >= SUMMARY_BATCH_SIZE:
        messages = history.messages
        total_count = len(messages)
        summarized_count = total_count - unsummarized

        # 获取未总结的消息（从summarized_count开始）
        new_messages = messages[summarized_count:]
        if new_messages:
            old_summary = _summary_store.get(self.__session_key, "")
            # 增量生成摘要信息
            new_summary = self.__generate_incremental_summary(
                old_summary,
                new_messages,
                llm)
            # 更新摘要
            _summary_store[self.__session_key] = new_summary
            # 重置未摘要消息数量
            _unsummarized_count[self.__session_key] = 0

            # 可选：清理已总结的消息，只保留摘要 + 最新一批消息
            # 这里采用软清理：保留最近 RECENT_WINDOW 轮完整消息
            # keep_count = RECENT_WINDOW * 2
            # if total_count > keep_count:
            #     # 保留最新的 keep_count 条消息
            #     history.messages = messages[-keep_count:]
```

#### 4.2.4. 事实facts如何管理？

事实是从对话中提取的**关键结构化信息**，用于长期保留那些在对话摘要压缩过程中可能丢失的重要数据。例如订单号、收货地址、联系电话、用户偏好等。

在lingyu的代码中，事实的存储结构是：

```pycharm
# 存储每个用户的事实(slots)：key是user_id:session_id，value是事实
_facts_store: Dict[str, Dict[str, str]] = {}
```

**为什么使用了压缩，还需要使用事实呢？**：对话摘要虽然能压缩历史，但存在两个问题：第一，摘要会不断被新内容覆盖，早期提取的订单号可能被“挤出去”；第二，摘要是自然语言描述，LLM需要重新理解才能提取信息，不如结构化数据直接可用。

事实存储解决了这两个问题：关键信息永久保留，且以结构化键值对形式存储，无需解析即可直接使用。

```python
# lingyu项目中，事实管理代码如下：

def get_key_facts(self) -> Dict[str, str]:
    """
    获取当前会话的关键事实
    :return: 关键事实字典
    键为事实名称, 值为事实值
    """
    return _facts_store.get(self.__session_key, {}).copy()

def update_key_facts(self, facts: Dict[str, str]) -> None:
    """
    更新当前会话的关键事实
    :param facts: 关键事实字典
    键为事实名称, 值为事实值
    """
    if self.__session_key not in _facts_store:
        _facts_store[self.__session_key] = {}

    _facts_store[self.__session_key].update({
        k: v for k, v in facts.items() if v
    })
    print("update key facts")


def clear_key_facts(self) -> None:
    """
    清除当前会话的关键事实
    """
    if self.__session_key in _facts_store:
        del _facts_store[self.__session_key]
```

#### 4.2.5. 如何给llm模块提供记忆？

LLM本身是**无状态**的，每次调用都需要主动传入历史信息。具体需要提供三个层次：

**第一，关键事实。** 包括订单号、地址、电话等结构化信息。这些信息在摘要压缩中容易丢失，需要独立存储并优先提供。

**第二，长期摘要。** 对早期对话的压缩总结，让LLM了解历史脉络而不必阅读全部消息。每6轮对话生成一次，控制在200字以内。

**第三，近期会话。** 最近6轮完整对话的原始内容，保留细节和上下文连贯性。

三者按“关键事实 → 长期摘要 → 近期会话”的顺序拼接后传入LLM，既控制Token成本，又保证记忆质量。

 ```python
 # 给llm提供记忆功能
 
 def prepare_memory_for_llm(self) -> List[BaseMessage]:
     history = self.get_session_history()
     messages = history.messages
 
     result = []
     # 1.添加关键事实
     facts = _facts_store.get(self.__session_key, {})
     if facts:
         facts_text = " | ".join([f"{key}: {value}" for key, value in facts.items()])
         result.append(AIMessage(content=f"[关键事实] {facts_text}"))
     # 2.添加摘要
     summary = _summary_store.get(self.__session_key, "")
     if summary:
         result.append(AIMessage(content=f"[摘要] {summary}"))
     # 3.添加新消息
     unsummarized_count = _unsummarized_count.get(self.__session_key, 0)
     if unsummarized_count > 0:
         recent_messages = messages[-unsummarized_count:]
         result.extend(recent_messages)
 
     return result
 ```

**流程说明**

**第一步**：用户输入消息，同时进入两个通道——意图识别模块分析用户想做什么，记忆管理模块提取历史信息。

**第二步**：意图识别输出意图（决定用什么策略回复）和槽位（提取出的订单号、地址等关键信息）。

**第三步**：根据意图选择对应的System Prompt，比如查物流用索要订单号的提示词，改地址用确认发货状态的提示词。

**第四步**：将System Prompt、记忆中的三层信息（事实、摘要、近期会话）、当前用户输入全部组装在一起。

**第五步**：调用LLM生成回答。

**第六步**：将本轮对话存入记忆，同时把意图识别提取的槽位更新到事实库。

**第七步**：返回回答给用户。
